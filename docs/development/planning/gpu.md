### From the compute (shipped 1.54.x) work
- TARGET SILICON (verified 2026-07-10, re-verified on iron at F0): archaemenid = Beelink SER, AMD Ryzen 7 5800H, Cezanne APU. iGPU at PCI `04:00.0`, device ID `1002:1638`. Compute ISA = **gfx90c** (GCN5 / Vega, IP version gfx9.3.0), fully documented in LLVM AMDGPU (gfx900/902/909/90c share the ISA doc). Display = **DCN 2.1**. Register aperture = **BAR5** MMIO. Renoir base-segment offsets are hardcoded tables (`renoir_ip_offset.h`, `soc15.c`) — there is NO IP-discovery walk on this part.
- C0 BOOT-STATE DUMP (iron, agnos 1.54.1, 2026-07-11) — the exact powered-on register values: `rlc=0x0` (RLC off), `me=0x15000000` (CP-gfx halted), `mec=0x50000000` (MEC halted), `psp=0x698e82` (PSP alive), GFX block powered. This is the reference cold-boot fingerprint for the GPU on this box.
- C1a0/C1a (agnos 1.54.2, iron): PSP **GPCOM ring is UP** — PSP-ready + ring-create + SETUP_TMR path works from agnos with no vendor driver.
- C1b-1 (agnos 1.54.4, iron): PSP `SETUP_TMR` returned `status=0x0`. This answered the memory-domain unknown: **low kernel physical memory IS PSP-DMA-reachable**, and the **TMR must live in the VRAM carveout**, placed via the **GFXHUB FB-location registers**.
- C1b-2 (agnos 1.54.5, iron, 2026-07-12): `LOAD_IP_FW` for **RLC_G** returned `status=0x0` — the FIRST sovereign-loaded firmware on the GPU; the PSP validated AMD's signature on a blob agnos handed it. This was the make-or-break round-trip of the whole compute thrust and it is DONE.
- C1c (agnos 1.54.6, iron, 2026-07-12): **ALL CP+MEC ucode loaded, 5/5** — CE / PFP / ME as whole-body loads, plus **MEC1 as a body+JT split**. **There is no MEC2 on gfx9.3.0** — do not go looking for it.
- C1d (agnos 1.54.7, iron, 2026-07-12): **CASE A reached — the GPU compute engine RUNS.** `gpu_engine_start()` un-halted CP-gfx (`me 0x15000000 -> 0`) and MEC1 (`mec 0x50000000 -> 0`) over an already-running RLC (`rlc=0x1`); pipe verified idle via GRBM bit31 = 0 and `stat=0x0`. Engine-start ORDER: RLC first (`RLC_CNTL` bit0), THEN clear the `CP_ME_CNTL` / `CP_MEC_CNTL` halts.
- FIRMWARE BLOB LAYOUT (CONFIRMED): the per-blob payload byte-range comes from `common_firmware_header` — `ucode_size_bytes` at offset **0x14**, `ucode_array_offset_bytes` at offset **0x18**.
- FIRMWARE POSTURE DECISION (user, 2026-07-11, standing): **ship blobs + a sovereignty tier.** Bundle the ~**628 KB** AMD-signed compute subset (`renoir_*.bin`) as hardware-required firmware (same posture as CPU microcode), PLUS a display-only zero-blob max-sovereignty tier. Claim wording stays precise: "sovereign except the vendor microcode." Firmware-provenance ledger belongs in docs. See [[project_firmware_blob_posture]].
- C2a GMC/GPUVM PROBE (agnos 1.54.8, iron): **GART is ABSENT** on this APU, and the FB carveout MC range is **`[0xF400000000, +3GB)`**. The resulting design — which is the shipped one — is: compute scratch lives in the UMA carveout and is addressed through the BIOS FB aperture with **ZERO page tables**, `VM_CONTEXT0` **disabled**. This deliberately designs the VM-fault-storm CPU-wedge failure mode out of existence.
- C2b (agnos 1.54.9, iron): GMC ARMED — L2 datapath on, paging off, the `0x1FFC` fault-net armed; CPU read/write of the carveout DRAM validated.
- C2c (agnos 1.54.10, iron): compute QUEUE MAPPED — **MEC1 / pipe0 / queue0** stood up via **direct (non-HWS) `CP_HQD` programming**. No HWS/KIQ scheduler is involved.
- C2d (agnos 1.54.12, iron): first PM4 packet FETCHED by the CP with `fault=0` — proving the ring GPUVA translates through the FB aperture with no page tables and no GART.
- C2e (agnos 1.54.13, iron): `WRITE_DATA` FENCE OK — the MEC executes a packet AND its `DST_SEL=5` memory write is **CPU-visible**, i.e. GPU→CPU coherence is proven on this path.
- C2f (agnos 1.54.16, iron): the **first hand-assembled gfx90c SHADER RAN** — a single-thread compute kernel executed on the shader cores and its `global_store_dword` result was read back on the CPU. Cost 3 burns.
- SUBMIT-SEQUENCE LAW (iron-proven, load-bearing): write the ring **WPTR LO before WPTR HI** — the CP latches on the HI write. Match the proven register-submit sequence byte-for-byte. This ordering gotcha, not the ISA, was the C2f bug.
- CACHE-PACKET ENCODING (settled 2026-07-22, corrected): `GPU_CP_COHER_CNTL_TCWB = 0x00840000` decodes as `TC_WB_ACTION_ENA` (1<<18) **PLUS** `TC_ACTION_ENA` (1<<23), and **bit 23 is the L2 INVALIDATE**. ### From the HDMI-audio physical model work
- HDMI PHYSICAL MODEL (durable, unique to this doc — nowhere else in agnos/docs): an HDMI link is a DVI link that steals the blanking. TMDS is always in one of three periods — Video Data (active pixels, 8b/10b), Control (filler), or Data Island (DIP). Audio + aux packets ride ONLY in horizontal/vertical blanking, guard-banded, TERC4-coded (NOT 8b/10b), BCH-protected. Consequence: video working over HDMI proves nothing about audio, because DVI and HDMI video are bit-identical.
- There is no 'I am HDMI' bit on the wire. DVI vs HDMI is BEHAVIORAL: the source carves data islands into blanking and fills them; the sink infers 'HDMI' by receiving valid data islands (above all the AVI InfoFrame). No islands -> sink stays in DVI receive mode with nowhere to route audio. The source is permitted to send islands because the sink's EDID carries the HDMI VSDB (IEEE OUI 0x000C03). The XB323U HAS the VSDB, so that gate was never agnos's blocker.
- Packet set required for stereo 48 kHz L-PCM, all sustained on cadence in the DIP: (a) Audio Sample Packet type 0x02 — the PCM, ~12,000 non-empty stereo ASPs/sec; `sample_present` selects which of 4 subpackets carry data; the B/block-start bit marks IEC-60958 channel-status boundaries; stereo = Layout 0. (b) Audio Clock Regeneration type 0x01 — sink regenerates 128*fs from TMDS via N/CTS: 128*fs = f_TMDS*(N/CTS); 48 kHz -> N=6144; at 241.5 MHz pixel clock -> CTS=241500; sent >=1/field. (c) Audio InfoFrame type 0x84 — CT/CC/CA metadata, >=1 per 2 fields; A SINK RECEIVING ASPs WITH NO AIF MUST MUTE; on DCN the AIF content is HW-generated from the Azalia CHANNEL_SPEAKER and the source only sets the SEND bit. (d) General Control Packet type 0x03 — carries AVMUTE; many sinks unmute only on an AVMUTE SET->CLEAR EDGE, not on a steady-cleared state (the vc4_hdmi upstream fix).
- The irreducible HDMI-audio bring-up ORDER, convergent across discrete TX chips (ADV7511 / IT66121 / SiI9022), vc4, i915, and pre-DMUB AMD DCE — and always executed as part of a MODESET, never as a live poke: (a) bring the port up and let TMDS settle / link-train FIRST (audio is downstream of a freshly-active link); (b) hold the audio path in reset while programming clock / N-CTS / format / channel-status / InfoFrame, then release on a clean EDGE; (c) SW writes N, HW derives CTS; (d) send AVI + Audio InfoFrame continuously; (e) schedule ASPs and CLEAR AVMUTE LAST (SET during setup -> CLEAR when ready = the edge the sink waits for). Discrete chips hide this behind one HDMI-mode bit that triggers an internal PHY transition + output bounce; AMD exposes the same transition as the DIG_ENABLE output-enable edge, the MMIO equivalent of ATOM TransmitterControl(ENABLE, HDMI).
- IRON PROOF, 2026-07-14 green-screen (the doc's single most load-bearing positive result, found nowhere else in the docs): when agnos briefly wrote amdgpu's YCbCr AVI bytes, the panel came up GREEN/PINK. ## ⚠ Standing iron hazards
Writes that blank, black or wedge the box; flags that must never be built; sequences with no recovery.

### From the compute (shipped 1.54.x) work
- **Stale-L2 trap.** Never CPU-pre-seed (UC) a buffer that the GPU is about to write and then use the read-back as an oracle — you get a ghost of the prior contents. Use a fresh, never-touched slot. This made the 1.54.18 burn unfalsifiable.
- **`ACQUIRE_MEM`/`GPU_CP_COHER_CNTL_TCWB = 0x00840000` INVALIDATES L2 (bit 23 `TC_ACTION_ENA`), it does not merely write back.** Any experiment that tries to *keep* L2 primed and then emits this packet destroys its own precondition. This exact mistake wasted the 1.56.4 burn (plan-S3 arm D).
- **WPTR submit order is load-bearing: LO then HI.** The CP latches on the HI write. Any other order = the CP never fetches, and the failure looks like a dead shader.
- **`mec_hdr != 0` is a broken ucode-presence guard** — the PSP path leaves the CP header-dump register at a `0xdefX_defX` sentinel (`0xdef0def0` observed), which defeats a `!= 0` test. Judge engine state on `CP_ME_CNTL`/`CP_MEC_CNTL` halt bits + `RLC_CNTL` + GRBM bit31 idle instead.
- **Engine-start ordering:** RLC first (`RLC_CNTL` bit0), then clear the CP-gfx and MEC halts (`me 0x15000000 -> 0`, `mec 0x50000000 -> 0`). Un-halting the CP over a dead RLC is not a tested path.
- **TMR placement:** the PSP TMR must be placed in the VRAM carveout via the GFXHUB FB-location registers. (Low kernel physical memory is PSP-DMA-reachable, which is what made `SETUP_TMR status=0x0` possible, but it is not where the TMR belongs.)
- **Byte-confirm every ISA/PM4 encoding before it rides a burn** — `llvm-mc -mcpu=gfx90c` as ground truth plus two adversarial re-derivations. The one time this doc asserted an encoding from memory (the `ACQUIRE_MEM` write-back claim) it burned a flash.
- **Diagnostic-ladder discipline: localize before fixing on iron.** No QEMU exists for AMD GFX/DCN, so every guess is a burn, and burns block the operator's only dev machine. Write each bite's driver code from the convergent prior art BEFORE the burn; the burn validates, it does not explore.
- **Build-size ceiling `scripts/check.sh:61` = `< 1,400,000` bytes**, with only 464 B of headroom as of 1.54.0. Compute register tables cross it. It is a discipline threshold, not a correctness bound — lift it rather than shrinking real code.

### From the display (shipped 1.55.x) work
- READ-HANG: reads of DCN registers behind a gated DISPCLK/DPPCLK domain, or in an unpowered pipe (OTG1-3 may be off), can HANG the box. Discipline that was proven safe: read `OTG_CONTROL` (0x1b41) FIRST and only touch HUBP registers for pipes that report enabled. Never blind-walk all HUBP instances on an un-probed part.
- BLACK-SCREEN PATHS on the `DIG_MODE` 2→3 (DVI→HDMI) commit — these three must be GATES INSIDE THE WRITER, never left to a separate burn to discover: (1) deep colour / scrambling enabled, (2) AVMUTE asserted, (3) any pixel format other than 24bpp RGB.
- SILENT-FAILURE ORDERING: writing `DTO0_MODULE` or `DTO_PHASE` before `DTO0_SOURCE_SEL`/`DTO_SEL` fails **silently** — hardware-enforced per AMD's `dce_audio.c` warning. No error, no log, just no audio. Always write source/select first.
- NEVER SNAP a measured display clock or refresh to a standard table or to the CVT 0.25 MHz grid. A snap can be **catastrophically wrong on an unlisted mode**, and on this exact link the grid fits a register-decode BUG 75x better than the truth. Refuse (cross-window spread > 200 ppm) rather than mis-snap — a silently-wrong DTO module is inaudible to test but wrong forever.
- NEVER derive raster timing, pixel clock or vblank period from `fb_width()`/`fb_height()` — those are the 800x600 SURFACE plane, not the 2560x1440 RASTER plane, and the DCN pipe scaler sits between them.
- QUIET-BOOT SCANOUT IS TILED/DCC: writing a LINEAR console into the GOP's surface with quiet-boot ON bands the glyphs (`fb_console.cyr:88-91`). Any new direct-FB writer must either honour the swizzle or run on agnos's own linear surface via the P1 re-point primitive.
- IRON-ONLY: QEMU emulates no AMD DCN. Every display bite must fall back cleanly to the direct-FB path off-iron; `gpu_display_probe` no-ops when the GPU/DCN is absent or the FB is not GOP-lit. Do not treat a green QEMU run as validation of any DCN write.

### From the shader (shipped 1.56.x) work
- MPCC_ALPHA_BLND_MODE mode 0 (PER_PIXEL) on the GOP's XRGB scanout surface, where the X byte is 0x00, makes the whole screen fully transparent = BLACK. Only mode 2 (GLOBAL_ALPHA), which ignores pixel alpha by definition, may ever be used against the GOP surface.
- `DOMAIN*_PG_CONFIG` sits at offset 0x0080 BASE_IDX 2 and NUMERICALLY COLLIDES with agnos's existing `GPU_R_OTG_PIXEL_RATE_CNTL = 0x80` at BASE_IDX 1. Writing with the wrong base POWER-GATES A LIVE PIPE. Name any new symbol distinctly (e.g. `GPU_R_DCPG_DOMAIN0_PG_CONFIG`) and assert the base in the comment.
- NEVER remove the pre-dispatch `ACQUIRE_MEM` invalidate (D-6). It is load-bearing twice: stale I$ on a rewritten shader VA, AND stale GL2 lines when a shader reads bytes CP-DMA just wrote MC-direct. S3 supplies the evidence (4096-of-4096 stale). The C2g-1 precedent for removing an 'obviously safe' cache op is EIGHT burns.
- There is NO GPU-side isolation: VM_CONTEXT0 paging is deliberately disabled, so a surplus lane's out-of-bounds store lands somewhere REAL in the carveout. Any deliberately-unguarded negative arm must be aimed at a sacrificial slot clear of the back buffers, the PSP TMR and the compute arena.
- DCN register writes must be INDEXED. A post-mortem defect in both D-lane tests: HUBPRET0 / MPCC0 / OPTC0 were written UNINDEXED while the surface-config register two lines away was correctly indexed.
- VOP3 and VOP3P take NO 32-bit literal on gfx9. Every selector/mask constant must be loaded with an `s_mov_b32` (SOP1) first. Named as the highest-probability first-burn bug in the packed path.
- There is NO scratch ring. `RSRC1`'s VGPR/SGPR fields must be recounted per kernel (`gpu_regs.cyr:1033-1035`); overflowing them is WRONG, not merely slow.
- `COMPUTE_PGM_LO = mc>>8` must stay exact with 256-byte-aligned resident slots (S12 multi-slot residency).
- Syscall #92 collides with Linux x86_64 `chown(path, uid, gid)`: arg1 is read as a user path pointer and arg2 (`len`) decodes as a uid, so a stray call on Linux is a METADATA WRITE if the path ever resolves. The file-level `#ifdef CYRIUS_TARGET_AGNOS` gate in `lib/syscalls.cyr` is load-bearing, and `scripts/agnos-crossbuild-gate.sh` coverage must include the row.
- Every new proof must use a FRESH, never-written arena slot pre-seeded with a sentinel distinct from both 0 and every expected value. Re-using a slot let a stale GL2 line FALSE-PASS a burn at C2g-1.
- A dispatch that 'completes' proves nothing, and neither does the done marker. Every oracle needs a lane/workgroup witness separating 'no store landed' from 'stored the wrong value' - the 1.54.17-19 lesson (the `nz` counter in `gpu_shader_dispatch2`).
- `s_mov_b64 exec,-1` must be issued FIRST on this raw non-HWS HQD path: the SPI hands the wave exec=0. `exec=-1` alone is correct ONLY for an exact 64-lane wave.
- A test whose oracle is a human looking at the screen must NEVER be a boot self-test - boot cannot ask a human anything. It belongs in a shell command / ring-3 binary. Five D-lane flashes would have been one.
- `gpu_blend_ok` gates NOTHING on the runtime path and MUST NOT - it is self-test-only (1.56.4 note). Do not wire it as a runtime gate by analogy with `gpu_matmul_ok`.
- A second full-screen 32bpp plane at 2560x1440@60 roughly DOUBLES scanout bandwidth against `DCHUBBUB_ARB_*` watermarks the GOP sized for ONE plane. Mitigation is a window-sized `RECOUT` (HUBP fetch scales with viewport - a 480x320 panel adds ~4% of a full plane).

### From the modeset / HDMI-audio investigation work
- DO NOT WRITE abs `0x52`. High-confidence analysis found bit8 is a pipe hold / soft-reset bracket around the WHOLE pipe (`0x110` open / `0x010` close in amdgpu's trace); latching it risks blanking the working scanout. It is NOT an audio lever, despite being one of the better-evidenced omissions in agnos's program.
- ATOM TRANSMITTER `#76` RUN LIVE BLANKS THE GOP PIPE NON-RECOVERABLY. It power-cycles the PHY (writes to `556F`, `5E03`, `5DF0`). Both PHYs are powered at boot, so there is NO 'safe' phyid — `#76` on EITHER phy is destructive. It stays behind `ATOM_RUN_TRANSMITTER` (default OFF) and is Linux-first only. Encoder-only `#4` is display-safe (and exonerated as silent).
- NO BLIND GPU REGISTER ACCESS — READS OR WRITES. Gated-clock domains hang on READ: a blind read-sweep of the OTG/DCCG blocks LOCKED archaemenid on 2026-07-17. Wrong PHY writes hang harder. Only header-verified offsets in known-ungated domains. NEVER a range sweep. (The one carved-out exception now proven safe is the `0x5Dxx`/`0x5Exx` PHY/RDPCS read domain.)
- The DMU/DMCUB region hangs on a blind read — use VERIFIED offsets only when touching `DMCUB_CNTL` / `DMCUB_SCRATCH*` / `DMCUB_INBOX1_*`. And a DMCUB soft-reset would tear down the GOP's DMCUB on this single-console box.

- A NEW `#ifdef` MODE-FLAG NEEDS ITS `build.sh` DEFINE LINE, AND YOU VERIFY IT LANDED BY `cmp`-ING THE TWO BINARIES, NOT BY TRUSTING THE BURN TAG. `ATOM_DRY` was a silent no-op for TWO burns — both 'live' and 'dry' built byte-identical and drove the PHY, blacking the display twice. Build only via `BURN_*=1 sh scripts/burn-prep.sh` and check the banner string.
- ATOM INTERPRETER LATENTS, cleared for #76/#4 but live for P6: (1) the recursion depth cap is 8 vs the amdgpu oracle's 32 — bump the cap AND the ws-pool level count TOGETHER if a P6 table nests deeper; (2) each ws slot is a fixed 512 B (128 dwords) but `ws_count` is a u8 (max 255), so a table with `ws_count > 128` overflows its slot into the next depth's — size slots to 1024 B (256 dwords) before running any large-workspace table.
- The operator's EAR is the only egress oracle (HDMI is transmit-only), and it is confounded by sink-latched state. Any audio A/B must be IN-BOOT (one flash, one sink state, one cable, one volume, one monitor power cycle) — cross-boot audio comparisons cannot control sink drift.
- Do not over-read a single amdgpu snapshot: `HDMI_CONTROL` bit3 differs between two audibly-playing captures (0716 vs 0720).

### From the HDMI-audio physical model work

Encoder-only table #4 (`DIGxEncoderControl` STREAM_SETUP) is display-safe (and exonerated as silent). #76 has already blanked this display twice.

## MODESET — standing rules

Each has already cost burns somewhere in the record.

- **⚠⚠ NO MODESET WRITE RUNS FROM BOOT.** Every write bite drives the hardware from `run /bin/modeset <phase>`
  (H3), never from an `#ifdef` at `main.cyr`. This is the D-lane lesson at its largest target: a blanked
  panel from a ring-3 tool costs a **power-button hold**; from a boot self-test it costs a **reflash**. It
  also gives phase-by-phase stepping with the camera already up, and N iterations per flash instead of one.
  ⚠ A burn carrying a new `/bin/modeset` needs **`install-media.sh --update-all`**, not `--update` (which is
  ESP-only). This has already bitten twice.
- **Anchor before you write. Never write a DERIVED DCN offset on the live pipe.** The 0x607 precedent: the
  arithmetically-correct pitch written to a mislabelled offset **under the OTG lock** produced HUBP underflow
  → black box → wedge. The 0x1275 precedent: a wrong offset that read a **plausible 0** (`MPCC_SM_CONTROL`,
  a real register at a valid address) instead of all-ones. Plausible ≠ correct; only a multi-constraint gate
  catches it.
- **⚠⚠ NEVER range-scan DCN, reads included.** A blind read-sweep of the OTG/DCCG blocks **locked
  archaemenid** on 2026-07-17. Named offsets only, in known-ungated domains, printed **progressively** so a
  hang localises to the last console line. "Reads are always safe" is FALSE for power-gated domains. (The
  0x5Dxx PHY domain is a *proven* exception for reads — that licenses nothing else.)
- **Index every DCN access by `gpu_display_pipe`.** The D-lane post-mortem found both tests writing
  `HUBPRET0` / `MPCC0` / `OPTC0` as **bare** offsets — those constants name instance 0 — while correctly
  striding the surface register two lines away. Harmless on a pipe-0 boot, wrong the moment a modeset touches
  another pipe. ⚠ **`gpu_dcn_write_committed` (`gpu.cyr:4977-4982`) has exactly this defect today**: it writes
  `gpu_reg32_write(GPU_BASE_DCN_2, off, val)` with **no pipe stride** while correctly striding the lock and
  the surface re-arm around it. Fix on adoption (M0).
- **A readback proves nothing.** Twelve mute burns of green gates came from echo registers, and on DCN a
  readback is also a *shadow*. Only three instruments in this arc cross that boundary: **I1**
  `OTG_STATUS_FRAME_COUNT` (0x1B4C `+ pipe*0x80`, field [23:0] mask 0xFFFFFF — counted scanned frames);
  **I2** PIT-counted refresh (`gpu_pixclk_discover`'s guts — an **off-GPU** time base, the only number in the
  display stack a register cannot fake); **I3** the ATOM `reads=/writes=/delays=` counters and the
  `ATOM_TRACE` write log, diffable against `atom-interp.py`. **Every bite must name which one it uses.**
- **State the two pictures in writing before the flash: "if X is true the screen/number shows ___, if false
  ___." If they are the same, redesign.** Four of the D lane's five burns were lost to harness defects, each
  fixing the previous cause and introducing a new unfalsifiable one.
- **One boot with COMPETING ARMS beats one hypothesis per burn.** D1 closed with an A/B/C/D/E phase burn; the
  SYMCLK bite closed with two off/on windows in one boot, eliminating sink drift, build variation and the
  version boundary at once.
- **Decode with a script, never by eye.** A D-lane burn was lost to reading 4294902881 as `0xFFFF0A61` when
  it is `0xFFFF0461`. Dumps print `rd <offset_dec> <value_dec>` and a **host-side** decoder names the fields.
- **Wall-clock holds only.** `gpu_dcn_hold(frames)` (`gpu.cyr:4986`) aborts the whole hold on one bounded-poll
  timeout and silently turned a 4-second hold into ~50 ms. Use `gpu_dcn_hold_ms(ms)` (`:5002`), which
  **returns its actual elapsed duration**, and print it.
- **Print `gpu_otg_lock_acked` at every lock site.** Burn 2 took the lock and printed nothing about it, so
  "committed under the lock" stayed an assumption through a whole wasted burn.
- **Read and save first; dead-man restore unconditionally; verify after.** Restore the value the GOP actually
  left — never a reconstructed default — because this pipe was initialised by firmware agnos did not write.
  The restore is not gated on the apply having worked or on any ack. Then confirm `OPTC0_UNDERFLOW` bit10
  (0x1ACA) is still 0 — an underflow is how a "successful" poke corrupts scanout without blanking it.
  ⚠ The bit is **sticky**: latch its entry value or the check is decorative.
- **Every `#ifdef` bite names its `BURN_*` flag in `burn-prep.sh`, gets its `build.sh` define line, and is
  proven by `cmp` PLUS a `verify_marker` row PLUS the NEGATIVE check** that the marker string is **absent**
  from a bare build. `cmp` proves the binaries differ, not that the difference is the code under test; this
  tree does not run DCE, so a marker in an always-compiled function verifies nothing. ⚠ `ATOM_DRY` shipped as
  a **silent no-op for two burns** — the flag existed in `burn-prep.sh` and had no `build.sh` define, so
  "DRY" built byte-identical to LIVE, drove the PHY and blacked the display **twice**. That defect class is
  still live: `burn-prep.sh:422-446` has rows for `HDA_HDMI`, `HDMI_ACR_CTS`, `ATOM_DRY`, `HDMI_ATOM` and
  fourteen others, and **no row for `ATOM_RUN_TRANSMITTER`, `ATOM_TRACE` or `ATOM_HALT`** — the transmitter
  flag being the one where absence-when-expected and presence-when-unexpected are both destructive. And a
  marker has already **drifted**: the 0718 log prints `atom: HDMI transmitter bringup OK`; `atom.cyr:864` now
  says `atom: HDMI bringup OK`.
- **The ear oracle needs a negative control in the same boot**, because sink amp/mute/mode state is
  **sink-latched** and every agnos instrument is source-side. "I believe I heard it" is not evidence — play a
  pattern the operator cannot guess, then ask for a match ([[feedback_ear_oracle_needs_negative_control]]).
- **⛔ Do not enable `BURN_AUDIO_TEARDOWN` / `gpu_hdmi_audio_disable` (`gpu.cyr:6972`) anywhere in this arc
  without deliberate intent.** The shutdown release pop is the arc's **only** sink-side instrument, and a
  clean teardown destroys it. The 1.55.20 ramp burn already demonstrated that class.
- **Name the bite ID and its governing D-rows before writing code, and re-open this document before each
  bite.** The 1.56.x S lane silently built the option D-3 explicitly rejected, across six bites, and the
  accompanying renumbering removed the means of noticing ([[feedback_execute_the_plan_you_wrote]]).
- **Check the burn record, not just the ladder.** A stale plan is worse than no plan when the work it
  describes is already done (decision D-7 of the shader arc; it cost a whole session three days ago).

### ⚠ H2 DISARM POLICY — DECIDED 2026-07-23, and it CONTRADICTS this plan's original text

**The kernel never disarms the latch. It is cleared by exactly one act: a human typing `rm /.modeset-armed`
at a prompt they can see.** That `rm` is intercepted in the `unlink` syscall's ext2 arm and routed to the
verified disarm (unlink → durable flush → re-lookup proving absence), so it cannot half-succeed.

The plan said *"delete it only after the pipe is verified still scanning (I1) and the shell is reached."*
Both conjuncts are unsound:

1. **The error costs are wildly asymmetric.** Disarm too early → a genuinely bad boot loses its latch, the
   next boot re-attempts, and the loop is **unbounded, invisible in the filesystem, and escapable only by
   reflash**. Disarm too late → a good boot fails to clear and the operator deletes one file. Too-late is
   bounded, visible and cheap. An automatic disarm optimizes for the expensive error.
2. **"Before `kybernet()`" is not "the shell was reached."** That point precedes the ELF load of
   `/bin/agnsh`, the `iretq` into ring 3 (the historical `#GP` site), and the fnptr paths that have hung
   before the banner. It means *the kernel is about to try*. Deleting the latch there and then wedging is
   strictly worse than shipping no latch at all.
3. **I1 cannot see a picture.** `gpu_display_wait_vblank` returns 1 the moment `OTG_FRAME_COUNT` changes,
   and the OTG times off its own PLL — HUBP underflow leaves it counting while the pipe emits black, and an
   out-of-range timing leaves it counting while the sink reports No Signal. Keying a disarm on I1 is the
   **echo-register** failure that cost twelve mute audio burns. I1 stays a logged datum, never a gate.
4. **The builds most likely to run a display experiment never reach the boot tail at all** —
   `AETHERSAFHA_SETU_SELFTEST`, `MISHRAN_DUPLEX_SELFTEST`, `TCP_LISTEN_SMOKE` and `ATOM_HALT` all end in an
   infinite idle. A manual disarm works in every build where a prompt exists.

**Cost of the decision:** one typed command per successful modeset boot, on a box the operator is already
standing in front of for a burn they initiated. **⚠ If anyone later adds an automatic disarm it becomes
MANDATORY to gate it on an armed-this-boot flag** (or a blocked boot clears a latch it never set), and the
only acceptable trigger is the first ring-3 blocking keyboard read — never the boot tail.

**Two more standing rules from this bite:**
- **Recovery is a COLD power-off** — hold the button until the fans stop, wait ~10 s, then start. Only a cold
  cycle re-runs AGESA/GOP and guarantees GPU/PSP re-init; the latch cannot help if the previous attempt left
  the silicon wedged across a warm reset.
- **Clear the latch before the boot that carries a FIX.** `install-media.sh --update` is ESP-only and never
  touches agnos-fs, so a new kernel arrives with the old latch intact and would be silently skipped.

**3D is not in this arc, and the argument is not size — it is recoverability.**

The two halves of "P6+" overlap in **zero** respects. Different silicon block: DCN 2.1 (DCE base segments,
HUBP/OTG/DIG/DCCG/MPC/AZ) versus GC (CP/SPI/PA/SC/VGT/CB/DB). Different register apertures and BASE_IDX —
with a documented numeric collision between them. Different foundation: the modeset rides `atom.cyr`, which
is **useless** for the graphics pipeline; 3D rides CP/PM4, which is **useless** for modeset. Different oracle:
a photograph and an ear versus a pixel diff. Different forcing function: the modeset has three live,
evidenced justifications; 3D has none.

**The decisive constraint: there is no GPU hang detection or recovery anywhere in agnos.** Every safety
property today is a *CPU-side* bounded watchdog (100 ms, `gpu_wait_done`) plus the `0x1FFC` VM fault net —
both bound how long the **CPU waits**, not what the **GPU does**. A wedged MEC does not stop scanout; that is
the entire reason the compute ring is survivable. **The GFX ring shares the GC block with everything, and the
GPU is also the console.** mabda's Cezanne bring-up notes (`mabda/src/backend_native_pm4.cyr:920-1100`) are a
catalogue of 2-second GFX TDRs — missing `SPI_SHADER_PGM_RSRC3_VS`/`LATE_ALLOC_VS`, missing guard-band
adjusts, `VGT_VERTEX_REUSE_BLOCK_CNTL` zero, binning at the APU default, missing `PFP_SYNC_ME`,
`CB_COLOR0_ATTRIB2` unset (the CB silently drops every pixel). **On Linux each of those was a 2-second
recovery and a retry. On agnos each is a dead box and a lost burn.**

**The size is also real.** A real RADV triangle on this exact Cezanne is **262 register writes** across 81
`SET_CONTEXT_REG` + 11 `SET_SH_REG` + 3 `SET_UCONFIG_REG_INDEX`. agnos's entire proven PM4 vocabulary is
**six packet types** and ~14 register writes per dispatch, with **zero** graphics registers defined anywhere
in the tree (verified by grep for `PA_SC|PA_CL|CB_COLOR|DB_Z_INFO|VGT_|SPI_SHADER|SET_CONTEXT|DRAW_INDEX|
CP_RB0|CLEAR_STATE`). Honest estimate: **25-40 iron bites across 3-4 minor versions**, on a box where the D
lane cost 5 burns for 1 result.

**The precedent is decisive.** Bite P5 of the 1.54.x plan said "**GFX-ring** 2D acceleration". What shipped at
1.55.30 was `PACKET3 DMA_DATA` on the existing MEC compute ring — "the FIRST hardware 2D on sovereign agnos …
no SDMA, no doorbell, no new firmware". **The GFX-ring prerequisite was imaginary one level down.**

"3D has no consumer, name it and it dissolves", is **DEAD** — killed by the 2026-07-23 correction at the top

## MODESET — definition of done

**The arc closes when agnos can take the pipe away from the GOP and give it back.** Concretely, all of:

1. **H0-H8 landed and green on host/QEMU.** Specifically: `klug_spill` proven by mounting the image from the
   host; the arm-once latch proven by a forced failure and a clean second boot; `/bin/modeset` staged and
   answering `--caps`; the ATOM instrument pack re-proving the 1.55.23 DRY trace byte-identically **and**
   returning non-zero on a deliberate desync; snapshot-DRY matching the oracle at a real branch path; the
   div32/mul32 sweep clean; the H7 table reproducing all ten anchors; `verify_marker` rows with negative
   checks for every ATOM and `MODESET_*` flag.
2. **L1 run and its answer recorded in `state.md`**, whichever way it fell.
3. **M1·M2·M3 captured** — the GOP-vs-amdgpu per-register delta at matched resolution, the PHY compared from
   agnos's own side, the scaler read for the first time — all committed to
   `agnosticos/docs/development/prior-art/`.
4. **M5 green:** agnos changed the raster and an **off-GPU instrument** measured the predicted change —
   59.951 Hz → 59.152 Hz, ±200 ppm, three orders of magnitude above the noise floor.
5. **★ M6 green: `OTG_MASTER_EN` 0 → reprogram → 1, with the frame counter observed stopping and resuming,
   the console intact, and underflow still 0.** This is the arc's stated closing condition. **agnos owns the
   OTG.**
6. **`gpu_otg_lock` / `gpu_otg_commit_unlock` / `gpu_dcn_write_committed` have real callers**, with the pipe
   stride fixed — the adopt-or-delete question at `gpu.cyr:5031-5036` answered by adoption.
7. **A4 has an answer, not a park.** Either M9 produces sound, or the arc records — with the modeset edge
   demonstrably performed — that **sequencing is eliminated** and A4's remaining candidates are (b) a write
   that doesn't latch and (c) the bare-metal environment, with L1's result naming the likelier.
8. **Docs true at close:** `state.md`'s A4 paragraph and `roadmap.md` rewritten (MD-1's blockers);
   `roadmap.md:131`'s "P6+ — 3D and full modeset" **split into two rows**, the modeset one closed and the 3D
   one pointing at a successor doc that does not exist yet and should not until a consumer is named; the
   dead `#ifdef HDMI_DCCG` block deleted; the burn ledger in `iron-nuc-zen-log.md` carrying a `#tracker-157x-*`
   section per M bite.
9. **Explicitly NOT required to close:** `SetPixelClock` #12 running live (MD-3); any mode change; the
   native-resolution console (MD-9); EDID/DDC/hotplug; DP anything; one line of 3D.
