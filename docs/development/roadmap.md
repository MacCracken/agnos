## 1.41.x — Shell Separation Arc (active)

**Goal**: move the interactive shell role out of the kernel. `kybernet` (PID 1) calls the in-kernel `shell()` (`kernel/user/shell.cyr` — was ~1150 LOC, **shrunk to 813 at 1.41.9**) — a ring-0 REPL that polls the keyboard directly (`kb_has_key`/`kb_read_scancode`), prints via `kprint`, and calls `ext2_*`/`vfs_*`/`fatfs_*` kernel functions in-process. The arc makes the **userland `agnsh` binary** (the [agnoshi](https://github.com/MacCracken/agnoshi) repo — AI-native NL shell, ~5 K LOC / 295 KB static, v1.3.x) the interactive shell, **exec'd from disk in ring 3** via the 1.40.x exec path. The in-kernel shell shrinks to a minimal **emergency/recovery fallback**. This is the first userland binary promoted to a system component, and it defines the **permanent kernel↔userland shell boundary** — the kernel-slimming counterpart to the 1.40.x exec arc (font→`kashi` already landed at 1.37.5).

**Why now**: 1.40.x exec-from-disk is iron-validated (a static ELF64 runs in ring 3 off the agnos-fs), and 1.40.14 process teardown means a long-lived userland process won't leak its slot/pages. `agnsh` is the natural first tenant.

**Gating prerequisite (cyrius-side): `CYRIUS_TARGET_AGNOS`.** `agnsh` is an OS-agnostic shell (zsh/bash-class) — the same source builds per Cyrius target, and the only targets today are `CYRIUS_TARGET_LINUX`/`_WIN`, so the current `build/agnsh` speaks the **Linux** syscall ABI and won't run on AGNOS's sovereign 28-syscall surface. The port is "add a Cyrius target" (a `lib/syscalls_*_agnos.cyr` stdlib peer emitting agnos syscall numbers), **not** "agnos answers Linux syscalls" (that would break the sovereign-surface invariant). This is **cyrius-gated**, driven with the cyrius agent — same class as PIE/TLS. The agnos-side bites below are necessary but not sufficient until that target lands; sequence both halves of the ABI together. **The frozen syscall contract both sides code against is [`agnos-userland-abi.md`](agnos-userland-abi.md)** (in this repo — normative kernel spec; the cyrius peer mirrors it). Boundary analysis: agnosticos [`shell-separation-prior-art.md`](https://github.com/MacCracken/agnosticos/blob/main/docs/development/shell-separation-prior-art.md) § 1a.

**The gap (current → required)**: a ring-3 `agnsh` can reach the kernel *only* through syscalls, but the in-kernel shell uses kernel internals directly. Two concrete surfaces are missing:
1. **Interactive stdin from ring 3.** `read(fd=0)` must return keyboard input the kernel services (today fd 0 → serial, `devs.cyr`). A long-lived interactive ring-3 process needs a **blocking stdin** the kernel drives from the keyboard IRQ — the first interactive extension of the run-to-completion exec model (the kernel enables interrupts in ring 0 while the syscall blocks, returns the byte/line to ring 3).
2. **Real FS syscalls.** `agnsh`'s builtins need `open`/`read`/`write`/`close` (exist) **plus** `getdents`/readdir, `mkdir`, `rmdir`, `unlink`, `rename`, `stat`, `chdir`/`getcwd` — `mkdir`/`rmdir` are tier-1 stubs returning 0 (`syscall.cyr`) and the rest don't exist as syscalls. They wire cleanly to the 1.40.13 mount-routed VFS backends.

**Bite ladder** (sequence, not commitments):
- **1.41.0 — arc open + boundary audit.** Inventory the in-kernel verb surface against the 28-entry syscall table; enumerate the syscall gaps `agnsh` needs; decide what stays in the emergency shell. Audit doc (agnosticos `shell-separation-prior-art.md`). Stage the `agnsh` static build onto the agnos-fs as `/bin/agnsh`.
- **1.41.1 — interactive stdin from ring 3.** Blocking `read(fd=0)` serviced from the keyboard driver (the syscall path enables IRQs in ring 0 while waiting), with backspace/line-edit parity with the in-kernel loop. The dispositive new capability — a userland process reads the keyboard for the first time.
- **1.41.2 — (parallel side-bite, NOT shell-separation) boot_info consumers.** gnoboot v0.5.0's sovereign `boot_info` fills got their agnos-side readers: a boot-time canary (reports `initramfs`/`cmdline`/`acpi_rsdp` — proves the gnoboot→agnos wire) + an ACPI RSDP fallback (`acpi_init` uses `boot_info+0x38` when the legacy BIOS scan finds nothing — the UEFI case). Took the next patch slot, shifting the shell-separation bites below down one. See CHANGELOG `[1.41.2]`.
- **1.41.3 — real FS syscalls. ✅ LANDED.** Nine syscalls wired to the mount-routed VFS against the frozen ABI: `open`(7) re-routed + flags, `mkdir`(9)/`rmdir`(10)/`sync`(12) made real, new `getdents`(29)/`unlink`(30)/`rename`(31)/`link`(32)/`stat`(33), plus the **`a4=r10`** ABI extension (decision O2). `chdir`/`getcwd` are NOT syscalls (ABI §3.2 — CWD is userland-owned). Validated by a kernel-side selftest (`fssys: ALL PASS`) + sweep 7/7. See CHANGELOG `[1.41.3]`. `agnsh`'s FS builtins now have a kernel face — but can't run until the cyrius `CYRIUS_TARGET_AGNOS` peer lands.
- **1.41.4 — `kybernet` execs `/bin/agnsh`. ✅ LANDED.** PID 1 `run`s the userland shell (`kybernet_exec_agnsh()` → `elf_load_from_file` → `exec_and_wait` → `proc_reap`) instead of calling in-kernel `shell()`; on exec failure (missing/corrupt `/bin/agnsh`) it falls back to the in-kernel emergency shell. **First boot-to-agnsh-on-disk** — unblocked by the **cyrius 6.0.56** toolchain leap (the `CYRIUS_TARGET_AGNOS` stdlib peer: `args_agnos`/`io`/`process_agnos` at 6.0.55/56), validated in QEMU by `scripts/agnsh-smoke.sh`: boot reaches the **agnsh prompt** in ring 3, no `#UD`, no emergency-shell fallback (PASS ×3). Iron burn pending. See CHANGELOG `[1.41.4]`.
- **1.41.5 — syscall-ingress hardening / security sweep. ✅ LANDED.** A dedicated hardening pass over the now-live ring-3→ring-0 surface (1.41.x put a real userland shell on the other side of the syscall boundary; 1.41.3 added nine FS syscalls taking user pointers). 15 of 26 adversarially-verified findings fixed: epoll_ctl/epoll_wait/timerfd fd **type-confusion** (arbitrary kernel write/read via a foreign fd's payload) closed with `VFS_*` tag checks + watched-fd bounds; `is_user_range`/`is_user_ptr` gained a **1 GB user-VA ceiling** (blocks aiming a kernel load/store at MMIO/high memory); the **ext2 basename overflow** of `ext2_dir_buf` capped at 255; `spawn` validates its ELF buffer; ELF `p_vaddr` upper-bounded; `sys_mmap` length-overflow capped; FAT-chain cycle guard. All behavior-preserving — sweep 7/7, `fssys: ALL PASS`, agnsh-smoke PASS. Audit: agnosticos [`kernel-1415-hardening-audit.md`](https://github.com/MacCracken/agnosticos/blob/main/docs/development/prior-art/kernel-1415-hardening-audit.md). See CHANGELOG `[1.41.5]`.
- **1.41.6 — refactor + exec-hardening continuation (the deferred half of the 1.41.5 sweep). ✅ LANDED.** Finishes the "refactoring" part of the hardening sweep, as its own validated cut. (1) A new `SYSCALL_HARDEN_SELFTEST` safety net (`shsys: ALL PASS`) regression-locks the 1.41.5 security fixes and covers the previously-untested epoll/timerfd/signalfd region. (2) **#11 ELF-leak teardown** — `elf_load` + `elf_load_from_file` validate every PT_LOAD segment in a **pre-pass before allocating** the address space, so a malformed ELF is rejected with zero allocations to leak (chosen over teardown in the iron-validated exec path). (3) **#22 dedup** — a shared `sc_path_ok` path-arg preamble across the 7 FS syscalls + collapsing the identical-body `FS_FAT`/`FS_EXFAT` arms to `||`. (4) **#21** — the trailing 1.41.3 FS group reordered into numeric order (29→33). #23/#14 canary uniformity documented (not a vuln; single-tail rejected as risky). All behavior-preserving — sweep 7/7, `fssys: ALL PASS`, `shsys: ALL PASS`, agnsh-smoke PASS. See CHANGELOG `[1.41.6]`.
- **1.41.7 — FAT/exFAT content-write via the syscall ABI. ✅ LANDED.** Discovered while planning the shell-shrink: `vfs_write` (the `write(fd)` syscall) was ext2-only, so FAT/exFAT content-write was reachable *only* from the in-kernel shell's `echo>` (`vfs_write_secondary`) — agnsh couldn't write a data partition, and the FAT/exFAT write tests were shell-coupled. Fixed first, so the shrink doesn't strand a capability: new `VFS_SEC_WFILE` write-fd — `open(7)` with write-access (`AO_WRONLY`/`AO_RDWR`) on a FAT/exFAT mount returns a fd that buffers content and flushes the whole file via `vfs_write_on` (→ `fatfs_write_file`/`exfat_write_file`) on close. Backed by a **static 4-slot pool freed on close** (not `kmalloc` — the alloc-only heap leaks + exhausts). New syscall-driven `syswr:` selftests land real content on FAT *and* exFAT disks, gated in fat-write-smoke + exfat-write-smoke. sweep 7/7, `fssys`/`shsys` PASS, agnsh-smoke PASS. Limits (follow-ons): 4 KB whole-file cap per fd, no `AO_APPEND`/streaming. See CHANGELOG `[1.41.7]`.
- **1.41.8 — decouple the FS-write test harness from the shell verbs. ✅ LANDED.** The migrate half of the shell-shrink, done first (migrate-then-delete safety): the FAT/exFAT/ext2 write **selftests** that drove the in-kernel shell verbs via `sh_exec("touch …")` now drive the write path via **syscalls** (new `sc_touch`/`sc_echo`/`sc_mkdir`/`sc_rmdir` helpers → `ksyscall`, with `strlen`-computed lengths so no hardcoded-length bugs). `rm`/`mv`/`sync`/`cat` stay shell verbs (recovery set). Same backend, identical on-disk files — fat/exfat/ext2-write smokes pass **unchanged** (sweep 7/7). The shell write verbs are now **undriven by any test**, so 1.41.9 can delete them with zero coverage loss. A multi-agent map + adversarial completeness spec drove this (caught an off-by-one path length the spec text itself got wrong). See CHANGELOG `[1.41.8]`.
- **1.41.9 — shrink the in-kernel shell to an emergency/recovery shell. ✅ LANDED.** Deleted the non-recovery verb handlers + dispatch + dead helpers: `touch`/`echo>`-redirect/`mkdir`/`rmdir`/`ln`/`bench`(+`bench_report`)/`ps`/`free`/`jbd2` + the now-dead `sh_write_selftest`. **`shell.cyr` 1149 → 813 LOC (−336)**; production kernel 1,084,312 → **1,070,288 B** (−14 KB). The recovery shell keeps `help`/`cd`/`pwd`/`ls`/`cat`/`run`/`mv`/`rm`/`sync`/`reboot` + the non-write verbs (uptime/lspci/cpus/net/parts/…); `echo` shrinks to a bare print (no redirect). The shared helpers (`sh_trim`/`sh_abspath`/`sh_split_parent`/…) stay — still used by the kept verbs. **The permanent kernel↔userland shell boundary is now locked**: kernel = recovery only, agnsh = full interactive. Validated — sweep 7/7 (the write smokes boot *into* the recovery shell), `fssys`/`shsys` PASS, agnsh-smoke PASS, recovery `agnos>` prompt comes up. Deletion done by a subagent from the 1.41.8 completeness spec, then independently verified (no dangling refs, all recovery verbs present). See CHANGELOG `[1.41.9]`.
- **1.41.10 — arc-close hardening. ✅ LANDED.** A multi-agent adversarial audit of the 1.41.x surface the 1.41.5 sweep predated — chiefly the never-audited **`VFS_SEC_WFILE` write-fd** (1.41.7) — surfaced 4 real findings (3 medium / 1 low, 0 refuted; the 1.41.5 regression-recheck came back clean). **3 fixed:** (#1) the write-fd's finite 4-slot pool leaked on process exit (`proc_reap`/exit never closed fds) → a ring-3 program could exhaust it and permanently DoS FAT/exFAT writes — fixed by an **fd-reclaim sweep in `proc_reap`** (close fds 3-31 of the exiting process; `vfs_close` is idempotent, safe under the single-foreground model); (#2) `vfs_sec_wfile_write` returned **0 at the 4 KB cap**, spinning a conforming ring-3 write loop forever → now returns an error; (#3) a **write-open of a directory** path flushed through `fatfs_write_file`/`exfat_write_file` and freed the dir's cluster chain (on-disk FS corruption) → both backends now refuse to overwrite a directory (attr 0x10 guard). All behavior-preserving — sweep 7/7, `fssys`/`shsys` PASS, agnsh-smoke PASS. The 1 low finding (#4, AO_CREAT rc ignored — fix not behavior-sound) rolls to 1.41.11. See CHANGELOG `[1.41.10]`.
- **1.41.11 — clean + iron-burn prep. ✅ LANDED.** The residual 1.41.10 low fix (audit #4): `vfs_close` now **surfaces the write-fd flush result** instead of swallowing it (a failed FAT/exFAT backend write — ENOSPC / write-protected — now reports the error to `close()` instead of silently dropping the data + reporting success; the audit's *primary* fix was unsound — it would have broken overwrite, since `fatfs_create` returns -1 on the normal name-collision case — so only the sound secondary fix was taken). Plus the iron-burn staging: a **1.41.x arc-burn tracker** ([agnosticos `#tracker-141x-cycle`](https://github.com/MacCracken/agnosticos/blob/main/docs/development/iron-nuc-zen-log.md#tracker-141x-cycle)) with the A1-A4 dispositive rubric (boot-to-agnsh / recovery-shell fallback / FAT-exFAT write / no-regression). All QEMU-green — sweep 7/7, `fssys`/`shsys` PASS, agnsh-smoke PASS. **The 1.41.x shell-separation arc is software-complete** — only the user-driven iron burn on archaemenid remains (the arc's first full hardware validation). See CHANGELOG `[1.41.11]`.

**Arc status: software-complete (1.41.1 → 1.41.11), iron burn PENDING (user-driven, staged at `#tracker-141x-cycle`).**

**Dependencies / honest caveats**:
- **Preemptive ring 3 (interrupt-KPTI) is NOT required** for the MVP separation, *as long as* `agnsh` runs its builtins via syscalls (a single long-lived ring-3 process making blocking syscalls fits the run-to-completion model once 1.41.1 lands). It **is** required for `agnsh` to launch *external* binaries (`agnsh` spawning a child program) without the run-to-completion constraint — that's a **follow-on** (ring-3-initiated `spawn`/`waitpid`), not part of this arc.
- **`agnsh`'s AI / intent / `hoosh`-gateway features are out of scope here** — the arc's bar is "agnsh is the interactive shell, from disk, talking to the kernel only via syscalls." LLM wiring (`hoosh`) is its own later arc.
- The in-kernel emergency shell is **permanent**, not transitional — it's the boot fallback when no userland shell is reachable. The boundary (kernel = recovery only; userland = full interactive) does not collapse.

## Full-Binary KASLR (Option A) — slot TBD on the cyrius v6.1.x PIE track

Reserved for whichever minor lands once cyrius PIE codegen ships. The data-KASLR shipped at 1.28.0 covers ~80% of the security value; Option A closes the last ~20% (gadgets pre-computed against the kernel binary itself, which currently sits at fixed `0x100000`).

**Hard prerequisite**: cyrius v6.1.x PIE codegen support. Filed at [cyrius/proposals/2026-05-11-pie-support.md](https://github.com/MacCracken/cyrius/blob/main/docs/development/proposals/2026-05-11-pie-support.md); slotted on the cyrius v6.x track after v6.0.0. When it lands, KASLR-Option-A work can begin in parallel with whatever 1.3x.y cycle is active; actual slot depends on cyrius ship cadence, not agnos's. agnos does **not** hand-roll a relocation table (rejected in `proposals/2026-05-11-kaslr-scope.md`).

**Work surface (when cyrius PIE is available):** boot shim grows ~2× (relocation walk + slid entry), kernel binary rebuilt with `--pie`, slide-aware crash-dump symbolizer, CI assertion rewrite (current `KASLR: pmm_next_free=N` probe stays; new `KASLR: kernel_slide=0x<hex>` probe lands alongside). Two-boot-diff assertion extended to cover the binary base.

**Pre-cyrius prep (no-op until PIE lands):**
- Audit remaining absolute-address assumptions in source (`proc_table` accessors, VFS slots, PciDev offsets already moved to named accessors — pre-existing wins that reduce the audit surface).
- Decide whether the slide range stays at 64 MB (boot-shim-friendly) or grows to full 4 GB (more entropy, more page-table work).
