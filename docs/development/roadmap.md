## 1.41.x тАФ Shell Separation Arc (тЬЕ iron-complete at 1.41.15 тАФ burn `14115`, 2026-06-06)

**Goal**: move the interactive shell role out of the kernel. `kybernet` (PID 1) calls the in-kernel `shell()` (`kernel/user/shell.cyr` тАФ was ~1150 LOC, **shrunk to 813 at 1.41.9**) тАФ a ring-0 REPL that polls the keyboard directly (`kb_has_key`/`kb_read_scancode`), prints via `kprint`, and calls `ext2_*`/`vfs_*`/`fatfs_*` kernel functions in-process. The arc makes the **userland `agnsh` binary** (the [agnoshi](https://github.com/MacCracken/agnoshi) repo тАФ AI-native NL shell, ~5 K LOC / 295 KB static, v1.3.x) the interactive shell, **exec'd from disk in ring 3** via the 1.40.x exec path. The in-kernel shell shrinks to a minimal **emergency/recovery fallback**. This is the first userland binary promoted to a system component, and it defines the **permanent kernelтЖФuserland shell boundary** тАФ the kernel-slimming counterpart to the 1.40.x exec arc (fontтЖТ`kashi` already landed at 1.37.5).

**Why now**: 1.40.x exec-from-disk is iron-validated (a static ELF64 runs in ring 3 off the agnos-fs), and 1.40.14 process teardown means a long-lived userland process won't leak its slot/pages. `agnsh` is the natural first tenant.

**Gating prerequisite (cyrius-side): `CYRIUS_TARGET_AGNOS`.** `agnsh` is an OS-agnostic shell (zsh/bash-class) тАФ the same source builds per Cyrius target, and the only targets today are `CYRIUS_TARGET_LINUX`/`_WIN`, so the current `build/agnsh` speaks the **Linux** syscall ABI and won't run on AGNOS's sovereign 28-syscall surface. The port is "add a Cyrius target" (a `lib/syscalls_*_agnos.cyr` stdlib peer emitting agnos syscall numbers), **not** "agnos answers Linux syscalls" (that would break the sovereign-surface invariant). This is **cyrius-gated**, driven with the cyrius agent тАФ same class as PIE/TLS. The agnos-side bites below are necessary but not sufficient until that target lands; sequence both halves of the ABI together. **The frozen syscall contract both sides code against is [`agnos-userland-abi.md`](agnos-userland-abi.md)** (in this repo тАФ normative kernel spec; the cyrius peer mirrors it). Boundary analysis: agnosticos [`shell-separation-prior-art.md`](https://github.com/MacCracken/agnosticos/blob/main/docs/development/shell-separation-prior-art.md) ┬з 1a.

**The gap (current тЖТ required)**: a ring-3 `agnsh` can reach the kernel *only* through syscalls, but the in-kernel shell uses kernel internals directly. Two concrete surfaces are missing:
1. **Interactive stdin from ring 3.** `read(fd=0)` must return keyboard input the kernel services (today fd 0 тЖТ serial, `devs.cyr`). A long-lived interactive ring-3 process needs a **blocking stdin** the kernel drives from the keyboard IRQ тАФ the first interactive extension of the run-to-completion exec model (the kernel enables interrupts in ring 0 while the syscall blocks, returns the byte/line to ring 3).
2. **Real FS syscalls.** `agnsh`'s builtins need `open`/`read`/`write`/`close` (exist) **plus** `getdents`/readdir, `mkdir`, `rmdir`, `unlink`, `rename`, `stat`, `chdir`/`getcwd` тАФ `mkdir`/`rmdir` are tier-1 stubs returning 0 (`syscall.cyr`) and the rest don't exist as syscalls. They wire cleanly to the 1.40.13 mount-routed VFS backends.

## Full-Binary KASLR (Option A) тАФ тЬЕ SHIPPED 1.47.4 (+ gnoboot 0.6.0)

The kernel binary's physical load base is now randomized per boot. The data-KASLR shipped at 1.28.0 covers ~80% of the security value; **Option A closed the last ~20%** тАФ the kernel binary no longer sits at fixed `0x100000`, so gadgets can't be pre-computed against it.

**What shipped:** the kernel builds as a relocation-free PIE (`CYRIUS_PIE=1` тЖТ ET_DYN, RIP-relative); gnoboot picks an RDRAND-slid, 2 MB-aligned base in [32 MB, 254 MB) and jumps to `base + e_entry`. The slide window sits above the 16 MB PMM ceiling + below the 256 MB per-proc-CR3 identity window, so **no kernel-side memory rework and no boot-shim relocation walk were needed**. Validated by `scripts/kaslr-smoke.sh` (two-boot base-diff) + exec-smoke 15/15 on a slid kernel; `KASLR: kernel_base=<hex>` boot probe.
