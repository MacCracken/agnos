## 1.41.x ‚Äî Shell Separation Arc (‚úÖ iron-complete at 1.41.15 ‚Äî burn `14115`, 2026-06-06)

**Goal**: move the interactive shell role out of the kernel. `kybernet` (PID 1) calls the in-kernel `shell()` (`kernel/user/shell.cyr` ‚Äî was ~1150 LOC, **shrunk to 813 at 1.41.9**) ‚Äî a ring-0 REPL that polls the keyboard directly (`kb_has_key`/`kb_read_scancode`), prints via `kprint`, and calls `ext2_*`/`vfs_*`/`fatfs_*` kernel functions in-process. The in-kernel shell shrinks to a minimal **emergency/recovery fallback**. This is the first userland binary promoted to a system component, and it defines the **permanent kernel‚Üîuserland shell boundary** ‚Äî the kernel-slimming counterpart to the 1.40.x exec arc (font‚Üí`kashi` already landed at 1.37.5).

**Why now**: 1.40.x exec-from-disk is iron-validated (a static ELF64 runs in ring 3 off the agnos-fs), and 1.40.14 process teardown means a long-lived userland process won't leak its slot/pages. `agnsh` is the natural first tenant.

**Gating prerequisite (cyrius-side): `CYRIUS_TARGET_AGNOS`.** `agnsh` is an OS-agnostic shell (zsh/bash-class) ‚Äî the same source builds per Cyrius target, and the only targets today are `CYRIUS_TARGET_LINUX`/`_WIN`, so the current `build/agnsh` speaks the **Linux** syscall ABI and won't run on AGNOS's sovereign 28-syscall surface. The port is "add a Cyrius target" (a `lib/syscalls_*_agnos.cyr` stdlib peer emitting agnos syscall numbers), **not** "agnos answers Linux syscalls" (that would break the sovereign-surface invariant). This is **cyrius-gated**, driven with the cyrius agent ‚Äî same class as PIE/TLS. The agnos-side bites below are necessary but not sufficient until that target lands; sequence both halves of the ABI together. **The frozen syscall contract both sides code against is [`agnos-userland-abi.md`](agnos-userland-abi.md)** (in this repo ‚Äî normative kernel spec; the cyrius peer mirrors it). Boundary analysis: agnosticos [`shell-separation-prior-art.md`](https://github.com/MacCracken/agnosticos/blob/main/docs/development/shell-separation-prior-art.md) ¬ß 1a.

**The gap (current ‚Üí required)**: a ring-3 `agnsh` can reach the kernel *only* through syscalls, but the in-kernel shell uses kernel internals directly. Two concrete surfaces are missing:
1. **Interactive stdin from ring 3.** `read(fd=0)` must return keyboard input the kernel services (today fd 0 ‚Üí serial, `devs.cyr`). A long-lived interactive ring-3 process needs a **blocking stdin** the kernel drives from the keyboard IRQ ‚Äî the first interactive extension of the run-to-completion exec model (the kernel enables interrupts in ring 0 while the syscall blocks, returns the byte/line to ring 3).
2. **Real FS syscalls.** `agnsh`'s builtins need `open`/`read`/`write`/`close` (exist) **plus** `getdents`/readdir, `mkdir`, `rmdir`, `unlink`, `rename`, `stat`, `chdir`/`getcwd` ‚Äî `mkdir`/`rmdir` are tier-1 stubs returning 0 (`syscall.cyr`) and the rest don't exist as syscalls. They wire cleanly to the 1.40.13 mount-routed VFS backends.

**Dependencies / honest caveats**:
- **Preemptive ring 3 (interrupt-KPTI) is NOT required** for the MVP separation, *as long as* `agnsh` runs its builtins via syscalls (a single long-lived ring-3 process making blocking syscalls fits the run-to-completion model once 1.41.1 lands). It **is** required for `agnsh` to launch *external* binaries (`agnsh` spawning a child program) without the run-to-completion constraint ‚Äî that's a **follow-on** (ring-3-initiated `spawn`/`waitpid`), not part of this arc.
- **`agnsh`'s AI / intent / `hoosh`-gateway features are out of scope here** ‚Äî the arc's bar is "agnsh is the interactive shell, from disk, talking to the kernel only via syscalls." LLM wiring (`hoosh`) is its own later arc.
- The in-kernel emergency shell is **permanent**, not transitional ‚Äî it's the boot fallback when no userland shell is reachable. The boundary (kernel = recovery only; userland = full interactive) does not collapse.

## Full-Binary KASLR (Option A) ‚Äî ‚úÖ SHIPPED 1.47.4 (+ gnoboot 0.6.0)

The kernel binary's physical load base is now randomized per boot. The data-KASLR shipped at 1.28.0 covers ~80% of the security value; **Option A closed the last ~20%** ‚Äî the kernel binary no longer sits at fixed `0x100000`, so gadgets can't be pre-computed against it.

**What shipped:** the kernel builds as a relocation-free PIE (`CYRIUS_PIE=1` ‚Üí ET_DYN, RIP-relative); gnoboot picks an RDRAND-slid, 2 MB-aligned base in [32 MB, 254 MB) and jumps to `base + e_entry`. The slide window sits above the 16 MB PMM ceiling + below the 256 MB per-proc-CR3 identity window, so **no kernel-side memory rework and no boot-shim relocation walk were needed**. Validated by `scripts/kaslr-smoke.sh` (two-boot base-diff) + exec-smoke 15/15 on a slid kernel; `KASLR: kernel_base=<hex>` boot probe.

**Correction to the analysis below:** the "hard prerequisite ‚Äî cyrius doesn't have a PIE mode" text predates the cyrius PIE landing (v6.1.6/6.1.8: `--pie` / `CYRIUS_PIE=1`, `EMITELF64_KERNEL` ET_DYN). That codegen turned out **relocation-free RIP-relative**, so the anticipated boot-shim relocation walk and the rejected hand-rolled relocation table were both moot.
