# 1.46.x SMP arc — sub-bite decomposition + the review's must-fix corrections

**Goal:** let APs (cores 1–3) run the scheduler concurrently with the BSP — flip `smp_wake_enabled=1` (main.cyr) and have each AP arm its LAPIC timer + run `sched_next`/`do_context_switch` under PER-CPU state with REAL locking, so `smp-smoke` shows `cpus online: 4` and the full battery stays green with 4 cores live.

**Source:** a 9-agent design+adversarial-review workflow (2026-06-24). The decomposition spine is sound (all 3 reviewers); the corrections below are mandatory and fold into the sub-bites they name.

**Safety invariant:** sub-bites 1–5 land while APs stay PARKED (`ap_entry` still `hlt`s), so single-core smokes stay green throughout. The wake flip is LAST and iron-gated. `smp_wake_enabled=0` fully reverts to the single-core MVP at any point.

## Sub-bites (ordered; never flip the wake until 1–6 are in + single-core-green)

1. **Restore the xchg spinlock primitive.** ✅ **DONE 2026-06-24** — `pmm_spin_lock`/`pmm_spin_unlock` (pmm.cyr) restored to the iron-proven xchg test-and-set (structured identically to smp.cyr `spin_lock`, no param → frame offsets verified byte-identical). FULLY single-core-validatable (lock always wins uncontended); `check.sh` 11/11, agnsh-smoke PASS. The `+58%` pmm cost is re-incurred deliberately as the foundation.
2. **Per-CPU scheduler identity.** ✅ **DONE 2026-06-24** — `proc_current` / `preempt_count` / `sched_idle_pid` → APIC-id-indexed arrays (`pcpu_curproc`/`pcpu_preempt`/`pcpu_idle`, `*_get()`/`*_set()` accessors) with a `cpu_count<2` single-core fast-path (slot 0, no APIC read → byte-for-byte the old globals). Global names removed → the forcing-function caught the full ~70-site set; build clean, `ring3-smoke` 8/8. **`kpti_kernel_cr3`/`kpti_user_cr3` were MOVED to sub-bite 4** — the syscall stub reads them via baked imm64, so they can't become arrays until the stub is regenerated (don't convert them in 2).
3. **Lock the shared global structures.** ✅ **heap_lock + vfs_lock DONE (2026-06-24).** `heap_lock` wired around `kmalloc` + `kfree_sized` (free-list RMW + `slab_grow`, which holds heap then takes `pmm_lock` → heap<pmm leaf-last; `slab_grow` traced to call only `pmm_alloc`/`vmm_map` → NO allocator re-entry). `vfs_lock` replaced `vfs_sec_wfile_alloc_slot`'s `preempt_disable`. **ROOT CAUSE of the earlier deadlock (my "re-entry" and "regalloc" theories were BOTH wrong):** the first wiring used a *parameterized* `spin_lock_at(addr)`, which was BROKEN — cyrius spills the param to the frame, shifting the `old`/`addr` locals off the asm's hardcoded `[rbp-N]` offsets, so it `xchg`'d **address 1** (a bogus low address) instead of the lock. It only "passed" for vfs because `[1]` happened to be 0; it DEADLOCKED in `kmalloc` once `[1]` became 1. The fixed-global `heap_spin_lock`/`vfs_spin_lock` pairs (NO param → frame identical to the proven `pmm_spin_lock` → correct offsets) fixed it. Root-caused via the QEMU monitor (`info registers`: RIP in the spin loop, RAX=1 = the xchg target). Validated: agnsh-smoke PASS, check.sh 11/11, ring3-smoke PASS, fat-write-smoke PASS. ✅ **proctab_lock DONE (2026-06-24).** `proc_alloc_slot`'s scan + claim + `proc_count`-bump now run under `proctab_lock` (fixed-global), AND the chosen slot gets a **CLAIMING sentinel (state=3)** before the lock releases — so a concurrent CPU's scan (which reuses only state==0) skips it until the caller writes state=1. States: 0=dead/free · 1=ready · 2=running · 3=claiming (sched_next selects only ==1). The reap-path `proc_count` decrements (`proc_reap`/`proc_reap_child`) are also under the lock. Inert single-core (the sentinel is overwritten before any scheduler runs); validated by ring3-smoke's non-LIFO-reuse + 8-proc-stress + thread-smoke (no regression). **Sub-bite 3 is COMPLETE for the global structures (heap/vfs/proctab).** `sched_lock` is NOT here — it folds into sub-bite 6/7 (it must wrap the pick+state-flip+save/restore, so it lives with the AP-runs-scheduler work). **NOTE (follow-on, sub-bite 5):** `proc_reap` sets state=0 BEFORE `proc_free_address_space` ("set before free"), so a concurrent alloc could grab a slot mid-teardown — the reap-vs-alloc teardown ordering (set state=0 AFTER the free, or hold the lock across) is deferred to the device/teardown locking pass; inert single-core.
4. **Per-CPU SYSCALL stub** (HARDEST) — per-CPU syscall kstack + SYSRET cells + kpti CR3 pair, computed from the live APIC id in the regenerated stub. Carries the **r10 reload hazard**.
5. **Lock FS scratch + device rings + console** — the breadth worklist (ext2/fatfs buffers, NVMe/AHCI/r8169/virtio rings, fb console, kb_buf).
6. **AP runs the scheduler** — rewrite `ap_entry` to arm its LAPIC timer + mask LVT + sti + idle-yield loop, but keep `smp_wake_enabled=0` so it's INERT (compiles, single-core trivially green).
7. **THE FLIP** — `smp_wake_enabled=1` + let `sched_next` hand real procs to APs. CANNOT be TCG-proven; iron-gated, with a `smp_sched_aps` kill-switch so STEP 1 (wake + AP-idle-only) burns separately from STEP 2 (real procs on APs).

## MANDATORY corrections from the adversarial review (apply in the named sub-bite)

- **[sub-bite 1] Lock order is heap < pmm, NOT pmm < heap.** `kmalloc` takes `heap_lock` then calls `pmm_alloc` (`pmm_lock`) — pmm is the LEAF, acquired innermost → HIGHEST in an acquire-low→high order. Re-derive the whole chain from actual call graphs. (Noted in pmm.cyr.)
- **[sub-bite 1/3] NEVER use a `spin_lock_at(addr)` with a parameter — CONFIRMED BROKEN, removed 2026-06-24.** The asm reads hardcoded `[rbp-0x10]`/`[rbp-0x08]`; cyrius spills the param to the frame and SHIFTS those locals. I wrongly dismissed this (claimed "the param doesn't shift, per cr3_load") — but the disassembly proved the `addr` local landed at `[rbp-0x18]` while the asm read `[rbp-0x10]` (which held `old`=1) → it `xchg`'d **address 1**. It locked a bogus low address: "passed" for vfs only because `[1]` was 0, DEADLOCKED in kmalloc once `[1]` became 1. **FIX (the reviewer's original recommendation): one FIXED-GLOBAL lock-fn pair per lock** (`heap_spin_lock`/`heap_spin_unlock`, `vfs_spin_lock`/`vfs_spin_unlock`), each structurally identical to the proven no-param `pmm_spin_lock` so the `[rbp-N]` offsets are correct. Done + validated.
- **[sub-bite 3, 2026-06-24] The `heap_lock` "deadlock" was NEVER re-entry OR regalloc — it was the broken `spin_lock_at` above (FALSIFIED both earlier theories).** I chased two wrong root causes: (1) "spinlock deadlocks on same-CPU re-entry" — built a recursive owner-aware lock, it still stalled; (2) "the call injection perturbs cc5 regalloc of the hot path." BOTH wrong. The QEMU monitor (`info registers` while hung: RIP in the spin loop, RAX=1) showed the `xchg` target was the literal value 1, i.e. the asm was reading the wrong frame slot — the parameterized `spin_lock_at` bug. With fixed-global `heap_spin_lock`, `kmalloc`+`kfree` under `heap_lock` boot clean (agnsh-smoke/check.sh/ring3/fat-write all PASS). **LESSON: when a hand-asm lock "works sometimes," disassemble + read registers at the hang FIRST — don't theorize about re-entry/regalloc from the source.**
- **[sub-bite 3, RESOLVED 2026-06-24] `proc_alloc_slot` needs a CLAIMING-STATE, not just a lock — DONE.** Wrapping it in `proctab_lock` alone fixes the append-count-bump race but NOT the dead-slot-reuse race: it returned an *unclaimed* slot (state still 0), so a second CPU's scan would re-grab the same `idx` before the caller set state. Fixed: under the lock, the chosen slot's state is set to a CLAIMING sentinel (**3** — ≠0 so the scan skips it, ≠1 so `sched_next` skips it, ≠2 so it isn't confused with running) before releasing; `proc_create`/`proc_create_full` overwrite it with state=1. State machine audited (0/1/2/3; nothing checks ==3 or ranges; `sched.cyr:200`'s "old≠0→ready" never touches a claiming slot since it's never the running `old`). Inert single-core, so validated as no-regression (ring3-smoke non-LIFO-reuse + 8-proc-stress + thread-smoke green).
- **[sub-bite 2] `proc_current` is ~69 sites, not ~30** — and includes NON-stub C readers the plan missed: **`ring3.cyr:114` reads `kpti_user_cr3` directly** and `ring3.cyr:62/109/110/139` read `proc_current`. The rename-to-compile-fail forcing-function catches these.
- **[sub-bite 2] APIC-id `>=4` clamp** — the hand-emitted stub `and eax,0xFF` has no `>=4 → 0` guard; a non-contiguous APIC topology (Zen multi-die/CCX) overruns the `[4]` arrays. Either assert contiguous 0..3 ids at `smp_start_aps` and refuse `id>=4`, or emit a clamp.
- **[sub-bite 4] The r10 reload IS mandatory** — keep the per-CPU stack-load strictly in rax/r9 (copy smp.cyr:172-188 register discipline), NEVER r10, AND reload `r10 = &kernel_rsp_save_pc + cpu*8` AFTER the stack switch before the consuming `mov rdi,[r10]`. **Read the APIC id ONCE at entry into a provably-dead scratch** (NOT rax=syscall#, NOT r9=a4, NOT r10) — do not re-read 0xFEE00020 7× per syscall, and do not reuse rax (it's the syscall number, shuffled to rsi). **Disassemble the generated `syscall_entry_buf` (objdump -b binary) before trusting it.**
- **[sub-bite 4] Stop the execwait#37 runtime stub-rebuild** — rewriting the shared `syscall_entry_buf` tears the instruction stream if another CPU executes it. Make the stub LOAD a per-CPU current-syscall-stack pointer (per-CPU primary+nested pair); make `ew37_busy` per-CPU or under `sched_lock`.
- **[sub-bite 4] Split `syscall_init`** into `syscall_msr_init` (per-CPU: STAR/LSTAR/SFMASK/EFER/IBRS — APs run this) and `syscall_stub_build` (BSP-ONCE: emits the shared buffer). APs must NOT re-emit the shared buffer.
- **[sub-bite 5/6] `timer_handler` MUST branch on `tss_get_current_cpu_id()`** — there is ONE handler on vector 32; today it unconditionally does `timer_ticks++` + `net_rx_drain()` + `do_context_switch`. Only CPU0 may advance `timer_ticks` + drain the NIC; APs run ONLY `do_context_switch` + `apic_eoi`. This is a CODE branch, not prose — it is the #1 TCG-invisible corruption (double-counted timebase + reentrant `net_rx_drain` corrupting `net_rx_pkt`). `net_poll_busy` stays the same-CPU flag (don't pretend it's a try-lock).
- **[sub-bite 4/6] Stack-layout collisions in region 1** — AP1's trampoline stack `0x310000-0x320000` abuts CPU1 TSS.RSP0 top `0x320000` (first interrupt grows into the trampoline). Move AP RSP0 stacks off the trampoline addresses (or retire trampoline stacks after check-in); carve per-CPU syscall kstacks from a clean pool (the bite-1 region-7 neighborhood) with a guard, NOT 0x360000-style addresses that may overlap.
- **[sub-bite 3/7] `sched_lock` must be held across the WHOLE pick+state-flip+save/restore** — and add a per-proc `on_cpu` field: `sched_next` skips any proc with `on_cpu != -1` (running/mid-switch elsewhere). Releasing `sched_lock` before the context restore lets two CPUs run the SAME proc.
- **[sub-bite 6/7] Gate each AP's scheduling on a PER-CPU ready flag** the AP sets AFTER creating its idle proc + initializing `proc_current_pc[my]`, NOT on the global `sched_active` (which the BSP flips at a moment unrelated to AP readiness).
- **[sub-bite 6] LVT mask is the LEADING-but-UNCONFIRMED keyboard-death fix** — mask AP LINT0 (0xFEE00350)/LINT1 (0xFEE00360). It's iron-only/QEMU-invisible; if it doesn't fix the regression, the flip regresses keyboard on iron → the `smp_sched_aps` kill-switch + STEP-1-alone burn is the mitigation. Also calibrate the AP LAPIC timer count per-CPU (don't blind-copy the BSP's QEMU-tuned count — real-Zen LAPIC bus differs).

## Sub-bite 4 — implementation design (2026-06-24, from reading the live stub)

The regenerated `syscall_entry_buf` has exactly **four baked addresses** that must become APIC-id-indexed:
1. `kpti_kernel_cr3` (emit @ syscall_hw.cyr:206 — `mov r10,imm64; mov r10,[r10]; mov cr3,r10`, VERY early)
2. `sc_entry_regs` (@222 — the rcx/r11/rbx/rbp/r12-15 capture base)
3. `kernel_rsp_save` (@244 — `mov r10,imm64; mov [r10],rsp`; **r10 then MUST survive to @317 `mov rdi,[r10]`** — the hazard)
4. `syscall_kstack_top` (@252 — `mov rsp,imm64`)

Plus `kpti_user_cr3` read by the EXIT path + `ring3.cyr:114` (a C reader, not the stub).

### VERIFIED via 10-agent adversarial review (2026-06-24) — Design 0 (APIC re-read), NOT a base-pointer register

The review (map → 2 designs → 6 adversarial verifies → synthesis) settled the approach and **corrected three of my own errors above**: `sc_entry_regs` stride is **8 cells / 64 B, not 9** (the stub captures exactly rcx/r11/rbx/rbp/r12-15 = 8 stores); the SIB disp32 must be emitted with **`store32(+4)`, NOT `store64(+8)`** (else every following instruction shifts → corruption); and KPTI is currently **collapsed** (kpti_kernel_cr3 == kpti_user_cr3 == running CR3) so the CR3 "switches" are same-value no-ops today.

**Chosen: Design 0 — carry cpu in r8 ONLY inside the entry window; RE-READ the APIC at the other two windows** (off-100 consumer → into rax; exit → into r8). 3 MMIO reads/syscall, **zero cross-call register-survival assumption**. (Design 1 parked the per-CPU base in r12 across `call syscall_handler`; cc5 *does* preserve r12 via frame-slot save/restore — so it'd work today — but it rests on a cross-kernel ABI invariant that rots as the kernel grows, and correction #27 says NEVER r12. Rejected.) **The r10 hazard is DISSOLVED**: point-3 SAVE becomes `mov [r8*8+disp32],rsp` (no r10), and the off-100 consumer re-reads the APIC into rax then `mov rdi,[rax*8+disp32]`.

**Per-CPU storage** (APIC-id-indexed, cpu capped <4 in BOTH the stub and every C accessor — `tss_get_current_cpu_id` does NOT cap): `pcpu_kpti_kernel_cr3[4]`, `pcpu_kpti_user_cr3[4]`, `pcpu_kernel_rsp_save[4]`, `pcpu_sc_entry_regs[32]` (8 u64 × 4, cpu c at byte c*64), `pcpu_syscall_kstack_top[4]` = `0xF10000+cpu*0x10000`, `pcpu_syscall_kstack_top2[4]` (nested) = `0xF90000+cpu*0x10000` — region 7 (PD[7]), above the proc_rsp0 0xF00000 ceiling, pmm-reserved, in the PD[0..127] copy so supervisor-mapped under every per-proc CR3. The `pcpu_*` .bss (currently ~0x197a08-0x223328, spans PD[0]+PD[1]) is reachable at point-1 under the still-user CR3 **because proc.cyr copies PD[0..127], NOT because it's <2 MB** (the "<2MB" reviewer claim was wrong). Build-assert all `&pcpu_* < 0x10000000` (PD[127]) and `< 0x7FFFFFFF` (disp32-positive).

**Decomposition — each value's stub edit AND ALL its C readers/writers land in the SAME bite** (GATE 1, both reviewers' #1 fatal: splitting a stub-writer from its C-reader = split-brain that fails the gated smokes):
- **4a** ✅ **DONE 2026-06-24** (objdump-verified, stub bytes UNCHANGED): split `syscall_init` → `syscall_msr_init` (APs run) + `syscall_stub_build` (BSP-once) + orchestrator. Instead of the global-`g_cpuid_edx` form (which needs a symbol address inside inline asm — uncertain mechanism), the `cpuid_edx [rbp-0x08]` hardcode was kept correct by making `cpuid_edx` `syscall_init`'s SOLE local (the proven `spin_unlock` sole-local pattern); objdump confirms the asm writes `%rdx,-0x8(%rbp)` and the C reads `-0x8(%rbp),%rax` — offsets match, IBRS intact. The `syscall_stub_built` once-guard was DEFERRED (it couples with the ew37 rebuild removal in 4b; the split alone gives APs `syscall_msr_init` without rebuilding, which is 4a's goal). agnsh/check11/ring3 green.
- **4b** (HIGHEST RISK — kstack VALUE moves 0x3F0000→0xF10000, must pass a real boot): per-CPU `kernel_rsp_save` + `syscall_kstack_top` in the stub (entry APIC→r8 block; point-3/point-4/off-100/exit rewrites) + ALL C sites (`sched.cyr:348`, ew37 snapshot `syscall.cyr:1120`/restore `:1248`) + REMOVE the ew37 runtime stub-rebuild (`syscall.cyr:1226-1227`).
- **4c**: per-CPU `kpti_kernel_cr3`/`kpti_user_cr3`/`sc_entry_regs` (stub points 1,2,6) + ALL C sites in lockstep via the delete-the-global compile-fail forcing function (`sched.cyr:221/378` + `:336-357`, `ring3.cyr:114`/`:83-84`, syscall.cyr ew37). The `+cpu*64` sc_entry_regs bias is TCG-VISIBLE (#44 yield breaks immediately if wrong).

**GATE 2 (every step):** `objdump -D -b binary -m i386:x86-64` the dumped buffer — confirm each `jb` lands EXACTLY past its `xor` (entry `jb +3` skips REX'd `45 31 C0`; off-100 `jb +2` skips `31 C0`), every disp32 leaves the next instruction decoding cleanly, `mov cr3,r10` = `41 0F 22 DA` (not `44…`→cr11→#UD), sc_entry_regs uses `shl r10,6`; 4a must be byte-IDENTICAL to pre-4a + `objdump -d` proves no surviving `[rbp-N]` for the cpuid result. Single-core agnsh/ring3/exec green at each step. **NO-GO without iron for any SMP-correctness claim** — TCG can't catch races; the cpu<4 cap only proves BSP-slot-0 byte-identity. ## Validation reality

TCG smokes CANNOT catch SMP races (scheduler selftests are TCG-only; KVM races kybernet at boot). So single-core green is NECESSARY but NOT SUFFICIENT for sub-bites 4/6/7 — those need careful-KVM (`-smp 4` + 2 procs spinning getpid/write, long dwell) + iron on archaemenid (the ONLY place keyboard-death + real-Zen INIT-SIPI timing manifest). Burn order via `smp_sched_aps`: STEP 1 (wake + AP-idle-only) first, THEN STEP 2 (real procs on APs). Read the framebuffer photo for iron diagnosis, never CMOS.
