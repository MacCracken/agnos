# 1.46.x SMP arc ‚Äî sub-bite decomposition + the review's must-fix corrections

**Goal:** let APs (cores 1‚Äì3) run the scheduler concurrently with the BSP ‚Äî flip `smp_wake_enabled=1` (main.cyr) and have each AP arm its LAPIC timer + run `sched_next`/`do_context_switch` under PER-CPU state with REAL locking, so `smp-smoke` shows `cpus online: 4` and the full battery stays green with 4 cores live.

**Safety invariant:** sub-bites 1‚Äì5 land while APs stay PARKED (`ap_entry` still `hlt`s), so single-core smokes stay green throughout. The wake flip is LAST and iron-gated. `smp_wake_enabled=0` fully reverts to the single-core MVP at any point.

## MANDATORY corrections from the adversarial review (apply in the named sub-bite)

- **[sub-bite 1] Lock order is heap < pmm, NOT pmm < heap.** `kmalloc` takes `heap_lock` then calls `pmm_alloc` (`pmm_lock`) ‚Äî pmm is the LEAF, acquired innermost ‚Üí HIGHEST in an acquire-low‚Üíhigh order. Re-derive the whole chain from actual call graphs. (Noted in pmm.cyr.)
- **[sub-bite 1/3] NEVER use a `spin_lock_at(addr)` with a parameter ‚Äî CONFIRMED BROKEN, removed 2026-06-24.** The asm reads hardcoded `[rbp-0x10]`/`[rbp-0x08]`; cyrius spills the param to the frame and SHIFTS those locals. I wrongly dismissed this (claimed "the param doesn't shift, per cr3_load") ‚Äî but the disassembly proved the `addr` local landed at `[rbp-0x18]` while the asm read `[rbp-0x10]` (which held `old`=1) ‚Üí it `xchg`'d **address 1**. It locked a bogus low address: "passed" for vfs only because `[1]` was 0, DEADLOCKED in kmalloc once `[1]` became 1. **FIX (the reviewer's original recommendation): one FIXED-GLOBAL lock-fn pair per lock** (`heap_spin_lock`/`heap_spin_unlock`, `vfs_spin_lock`/`vfs_spin_unlock`), each structurally identical to the proven no-param `pmm_spin_lock` so the `[rbp-N]` offsets are correct. Done + validated.
- **[sub-bite 3, 2026-06-24] The `heap_lock` "deadlock" was NEVER re-entry OR regalloc ‚Äî it was the broken `spin_lock_at` above (FALSIFIED both earlier theories).** I chased two wrong root causes: (1) "spinlock deadlocks on same-CPU re-entry" ‚Äî built a recursive owner-aware lock, it still stalled; (2) "the call injection perturbs cc5 regalloc of the hot path." BOTH wrong. The QEMU monitor (`info registers` while hung: RIP in the spin loop, RAX=1) showed the `xchg` target was the literal value 1, i.e. the asm was reading the wrong frame slot ‚Äî the parameterized `spin_lock_at` bug. With fixed-global `heap_spin_lock`, `kmalloc`+`kfree` under `heap_lock` boot clean (agnsh-smoke/check.sh/ring3/fat-write all PASS). **LESSON: when a hand-asm lock "works sometimes," disassemble + read registers at the hang FIRST ‚Äî don't theorize about re-entry/regalloc from the source.**
- **[sub-bite 3, RESOLVED 2026-06-24] `proc_alloc_slot` needs a CLAIMING-STATE, not just a lock ‚Äî DONE.** Wrapping it in `proctab_lock` alone fixes the append-count-bump race but NOT the dead-slot-reuse race: it returned an *unclaimed* slot (state still 0), so a second CPU's scan would re-grab the same `idx` before the caller set state. Fixed: under the lock, the chosen slot's state is set to a CLAIMING sentinel (**3** ‚Äî ‚âÝ0 so the scan skips it, ‚âÝ1 so `sched_next` skips it, ‚âÝ2 so it isn't confused with running) before releasing; `proc_create`/`proc_create_full` overwrite it with state=1. State machine audited (0/1/2/3; nothing checks ==3 or ranges; `sched.cyr:200`'s "old‚âÝ0‚Üíready" never touches a claiming slot since it's never the running `old`). Inert single-core, so validated as no-regression (ring3-smoke non-LIFO-reuse + 8-proc-stress + thread-smoke green).
- **[sub-bite 2] `proc_current` is ~69 sites, not ~30** ‚Äî and includes NON-stub C readers the plan missed: **`ring3.cyr:114` reads `kpti_user_cr3` directly** and `ring3.cyr:62/109/110/139` read `proc_current`. The rename-to-compile-fail forcing-function catches these.
- **[sub-bite 2] APIC-id `>=4` clamp** ‚Äî the hand-emitted stub `and eax,0xFF` has no `>=4 ‚Üí 0` guard; a non-contiguous APIC topology (Zen multi-die/CCX) overruns the `[4]` arrays. Either assert contiguous 0..3 ids at `smp_start_aps` and refuse `id>=4`, or emit a clamp.
- **[sub-bite 4] The r10 reload IS mandatory** ‚Äî keep the per-CPU stack-load strictly in rax/r9 (copy smp.cyr:172-188 register discipline), NEVER r10, AND reload `r10 = &kernel_rsp_save_pc + cpu*8` AFTER the stack switch before the consuming `mov rdi,[r10]`. **Read the APIC id ONCE at entry into a provably-dead scratch** (NOT rax=syscall#, NOT r9=a4, NOT r10) ‚Äî do not re-read 0xFEE00020 7√ó per syscall, and do not reuse rax (it's the syscall number, shuffled to rsi). **Disassemble the generated `syscall_entry_buf` (objdump -b binary) before trusting it.**
- **[sub-bite 4] Stop the execwait#37 runtime stub-rebuild** ‚Äî rewriting the shared `syscall_entry_buf` tears the instruction stream if another CPU executes it. Make the stub LOAD a per-CPU current-syscall-stack pointer (per-CPU primary+nested pair); make `ew37_busy` per-CPU or under `sched_lock`.
- **[sub-bite 4] Split `syscall_init`** into `syscall_msr_init` (per-CPU: STAR/LSTAR/SFMASK/EFER/IBRS ‚Äî APs run this) and `syscall_stub_build` (BSP-ONCE: emits the shared buffer). APs must NOT re-emit the shared buffer.
- **[sub-bite 5/6] `timer_handler` MUST branch on `tss_get_current_cpu_id()`** ‚Äî there is ONE handler on vector 32; today it unconditionally does `timer_ticks++` + `net_rx_drain()` + `do_context_switch`. - **[sub-bite 4/6] Stack-layout collisions in region 1** ‚Äî AP1's trampoline stack `0x310000-0x320000` abuts CPU1 TSS.RSP0 top `0x320000` (first interrupt grows into the trampoline). - **[sub-bite 3/7] `sched_lock` must be held across the WHOLE pick+state-flip+save/restore** ‚Äî and add a per-proc `on_cpu` field: `sched_next` skips any proc with `on_cpu != -1` (running/mid-switch elsewhere). Releasing `sched_lock` before the context restore lets two CPUs run the SAME proc.
- **[sub-bite 6/7] Gate each AP's scheduling on a PER-CPU ready flag** the AP sets AFTER creating its idle proc + initializing `proc_current_pc[my]`, NOT on the global `sched_active` (which the BSP flips at a moment unrelated to AP readiness).
- **[sub-bite 6] LVT mask is the LEADING-but-UNCONFIRMED keyboard-death fix** ‚Äî mask AP LINT0 (0xFEE00350)/LINT1 (0xFEE00360). It's iron-only/QEMU-invisible; if it doesn't fix the regression, the flip regresses keyboard on iron ‚Üí the `smp_sched_aps` kill-switch + STEP-1-alone burn is the mitigation. ### VERIFIED via 10-agent adversarial review (2026-06-24) ‚Äî Design 0 (APIC re-read), NOT a base-pointer register

The review (map ‚Üí 2 designs ‚Üí 6 adversarial verifies ‚Üí synthesis) settled the approach and **corrected three of my own errors above**: `sc_entry_regs` stride is **8 cells / 64 B, not 9** (the stub captures exactly rcx/r11/rbx/rbp/r12-15 = 8 stores); the SIB disp32 must be emitted with **`store32(+4)`, NOT `store64(+8)`** (else every following instruction shifts ‚Üí corruption); and KPTI is currently **collapsed** (kpti_kernel_cr3 == kpti_user_cr3 == running CR3) so the CR3 "switches" are same-value no-ops today.

**Chosen: Design 0 ‚Äî carry cpu in r8 ONLY inside the entry window; RE-READ the APIC at the other two windows** (off-100 consumer ‚Üí into rax; exit ‚Üí into r8). 3 MMIO reads/syscall, **zero cross-call register-survival assumption**. (Design 1 parked the per-CPU base in r12 across `call syscall_handler`; cc5 *does* preserve r12 via frame-slot save/restore ‚Äî so it'd work today ‚Äî but it rests on a cross-kernel ABI invariant that rots as the kernel grows, and correction #27 says NEVER r12. Rejected.) **The r10 hazard is DISSOLVED**: point-3 SAVE becomes `mov [r8*8+disp32],rsp` (no r10), and the off-100 consumer re-reads the APIC into rax then `mov rdi,[rax*8+disp32]`.

**Per-CPU storage** (APIC-id-indexed, cpu capped <4 in BOTH the stub and every C accessor ‚Äî `tss_get_current_cpu_id` does NOT cap): `pcpu_kpti_kernel_cr3[4]`, `pcpu_kpti_user_cr3[4]`, `pcpu_kernel_rsp_save[4]`, `pcpu_sc_entry_regs[32]` (8 u64 √ó 4, cpu c at byte c*64), `pcpu_syscall_kstack_top[4]` = `0xF10000+cpu*0x10000`, `pcpu_syscall_kstack_top2[4]` (nested) = `0xF90000+cpu*0x10000` ‚Äî region 7 (PD[7]), above the proc_rsp0 0xF00000 ceiling, pmm-reserved, in the PD[0..127] copy so supervisor-mapped under every per-proc CR3. ## Validation reality

TCG smokes CANNOT catch SMP races (scheduler selftests are TCG-only; KVM races kybernet at boot). So single-core green is NECESSARY but NOT SUFFICIENT for sub-bites 4/6/7 ‚Äî those need careful-KVM (`-smp 4` + 2 procs spinning getpid/write, long dwell) + iron on archaemenid (the ONLY place keyboard-death + real-Zen INIT-SIPI timing manifest). Burn order via `smp_sched_aps`: STEP 1 (wake + AP-idle-only) first, THEN STEP 2 (real procs on APs). Read the framebuffer photo for iron diagnosis, never CMOS.
